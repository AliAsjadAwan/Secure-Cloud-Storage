Problem Statement

💰

🔓

Plaintext Storage

Most cloud providers store data with server-side keys they control

⚠️

No Integrity Check

Files can be tampered with during storage or transit undetected

→ We need client-controlled encryption + integrity verification + access control

Research Objectives

  • Implement AES encryption (via Fernet) for files before cloud upload
  • Implement SHA-256 integrity verification on every download
  • Build secure authentication with Bcrypt, rate limiting, and account lockout
  • Implement role-based access control (RBAC) for authorization
  • Create HMAC-signed tokens for time-limited download links
  • Implement audit logging for security monitoring
  • Deploy on production cloud with HTTPS enforcement

Encryption Workflow

📤 Upload (Encrypt)

  • User selects file
  • Magic-byte MIME validation
  • SHA-256 hash computed & stored
  • Fernet.encrypt() — AES-128-CBC
  • Encrypted blob → Cloudinary
  • Metadata → PostgreSQL

📥 Download (Decrypt)

  • HMAC token generated (5-min expiry)
  • Token signature + expiry verified
  • Ownership/share permission check
  • Encrypted blob fetched from cloud
  • Fernet.decrypt() — verify HMAC tag
  • SHA-256 hash compared → serve file

Encryption Implementation

# Fernet encryption (AES-128-CBC + HMAC-SHA256)
from cryptography.fernet import Fernet

encryption_key = os.getenv("FERNET_KEY").encode()
cipher = Fernet(encryption_key)

# UPLOAD: Encrypt before cloud storage
data = file.read()
file_hash = hashlib.sha256(data).hexdigest()     # Integrity hash
encrypted_data = cipher.encrypt(data)             # AES encrypt
cloudinary.uploader.upload(BytesIO(encrypted_data), resource_type='raw')

# DOWNLOAD: Decrypt and verify integrity
encrypted_data = requests.get(file.storage_url).content
decrypted = cipher.decrypt(encrypted_data)        # AES decrypt + HMAC verify
current_hash = hashlib.sha256(decrypted).hexdigest()

if current_hash != file.file_hash:                # Integrity check
    return "File integrity compromised!"

Authentication & Access Control

🔑 Authentication

  • Bcrypt password hashing with auto-salt
  • Rate limiting: 5 requests/min on login
  • Account lockout: after 5 failed attempts
  • Password policy: 8+ chars, upper, lower, digit, special
  • Secure cookies: HttpOnly + Secure + SameSite=Lax

👥 Authorization (RBAC)

  • User — Manage own files, share, view analytics
  • Admin — View all files, users, audit logs, stats
  • Ownership checks on every file operation
  • Share permissions: view / download with expiry
  • Revocation: instant access termination

Security Hardening

🛡️

CSRF Protection

Flask-WTF generates unique tokens per session. Every POST verified server-side.

📋

Content Security Policy

Restricts scripts, styles, fonts to trusted origins. Blocks inline script execution.

🔒

HSTS

Forces HTTPS with 1-year max-age. Prevents protocol downgrade attacks.

🚫

Malware Blocking

Magic-byte analysis + dangerous signature detection (PE, ELF, shebangs).

⏱️

Token Expiry

HMAC-signed download tokens expire after 5 minutes. No replay possible.

📊

Audit Logging

Every login, upload, download, share, and revocation is logged with timestamps.

Limitations & Future Work

⚠️ Current Limitations

  • Server has access to plaintext during encryption
  • Single Fernet key for all users
  • No two-factor authentication
  • No file versioning

🚀 Future Enhancements

  • End-to-End Encryption — Web Crypto API
  • 2FA — TOTP via Google Authenticator
  • Key Rotation — Automated re-encryption
  • Zero-Knowledge — Client-side key derivation
  • GDPR/HIPAA — Compliance features

Conclusion

We designed and deployed a production-ready secure cloud storage system that encrypts files with AES-128-CBC before cloud upload, verifies integrity with SHA-256, protects credentials with Bcrypt, and controls access through RBAC and HMAC-signed time-limited tokens.
